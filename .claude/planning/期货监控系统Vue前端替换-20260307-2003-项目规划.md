# 项目规划：期货监控系统 Vue 前端替换

## 1. 背景与目标

### 1.1 要解决的问题
当前系统使用 Tkinter 桌面界面，已具备真实行情监控与策略能力，但在专业化 UI、可部署性、前后端解耦方面存在上限。用户目标是替换为 Vue 前端，并保持现有策略/行情能力不变。

### 1.2 目标与成功标准
- 用 Vue 前端替换 Tkinter 前端，形成标准 Web 看板。
- 后端服务化（HTTP + WebSocket），复用现有策略引擎。
- 保留真实行情模式（strict real mode），不回退 mock。
- 保留 ALL 输入能力，并按业务范围处理“全部品种”。
- 能对接现有自动部署流程，域名为 `kline.godpenai.com`。

成功标准：
1. 前端可完成 启动/停止/标记已买入/查看日志/查看品种状态。
2. 后端稳定推送实时事件（row/log/running/connection）。
3. ALL 模式可正常工作，且不出现“ALL伪品种行”。
4. 本地可运行，生产可接入自动部署链路。

---

## 2. 关键决策与依据（Why）

### 2.1 后端服务形态

**候选方案**
1) FastAPI + WebSocket
2) FastAPI + 纯 HTTP 轮询
3) 保持 Tkinter + 少量接口

**最终选择**：FastAPI + WebSocket（用户已确认）

**选择理由**
- 与现有 Python 策略代码同栈，重用成本最低。
- 实时监控场景下，WebSocket 比轮询更稳定且更省资源。
- API 文档和契约可标准化，便于后续迭代。

**不选其他方案原因**
- 纯轮询在高频更新和多品种场景下成本高、实时性差。
- 继续 Tkinter 不满足“专业标准化 Vue 前端”目标。

---

### 2.2 前端技术栈

**候选方案**
1) Vue3 + Vite + TypeScript + Pinia + Element Plus
2) Vue3 + Naive UI
3) React 技术栈

**最终选择**：Vue3 + Vite + TS + Pinia + Element Plus

**选择理由**
- 与用户“Vue 前端”要求完全一致。
- Element Plus 对企业级表格、表单、状态展示成熟度高。
- Pinia 易于管理实时状态流（WS 事件 -> Store -> 组件）。

**不选其他方案原因**
- Naive UI 也可行，但团队模板与社区范式覆盖度略弱。
- React 与用户明确要求不一致。

---

### 2.3 ALL 模式品种范围

**候选方案**
1) 仅国内期货交易所（CFFEX/SHFE/DCE/CZCE/INE/GFEX）
2) 全市场可查询合约（含外盘映射）
3) 自定义白名单品种组

**最终选择**：仅国内期货交易所（用户已确认）

**选择理由**
- 与项目业务边界一致（国内期货监控）。
- 减少噪音数据，降低前端压力和误判风险。
- 对“ALL=全部”的用户心智更清晰：全部国内期货有效合约。

**不选其他方案原因**
- 全市场会引入外盘/映射品种，偏离当前业务。
- 白名单维护成本高，不适合作为默认模式。

---

### 2.4 部署策略

**候选方案**
1) 前后端分离 + Nginx 反代
2) 后端一体托管前端静态文件
3) Docker Compose

**最终选择**：本阶段不锁死部署方式，按现有自动部署链路对接（用户已明确“部署时候再看”）

**选择理由**
- 用户已有自动化部署体系，当前最优是先交付“可接入”的标准产物：
  - 前端 `dist/`
  - 后端 ASGI 服务入口
  - 明确的 `/api` 与 `/ws` 路由契约
- 域名已确定为 `kline.godpenai.com`，部署阶段重点是网关路由与证书配置。

**不选其他方案原因**
- 现在提前锁定具体部署工具会和用户既有自动部署体系冲突。

---

### 2.5 架构改造策略（增量重构）

**候选方案**
1) 在现有逻辑上做增量服务化（提取 MonitorService）
2) 全量重写监控引擎
3) 仅加 API 包装，不整理核心逻辑

**最终选择**：增量服务化

**选择理由**
- 现有策略代码已跑通，增量重构风险最小。
- 可复用 `market.py / breakout.py / state_machine.py / storage.py`。
- 保证行为一致性，降低回归成本。

**不选其他方案原因**
- 全量重写风险高且不必要。
- 仅薄包装会让后续维护困难。

---

## 3. 实施范围与阶段划分

### 3.1 本期范围
1. 后端服务化：
   - 新增 API 层（配置、监控控制、状态查询）
   - 新增 WS 事件推送
   - 监控服务与事件总线抽象
2. Vue 前端：
   - 监控总览页（表格、状态、日志）
   - 配置与控制区（symbols/ALL、启动停止、标记已买入）
3. ALL 国内期货范围约束：
   - ALL 解析后过滤到国内期货交易所
4. 运行与部署交付：
   - 本地一键运行脚本
   - 可对接自动部署的构建产物与环境变量说明

### 3.2 非本期范围
- 自动下单与交易执行。
- 新策略开发（仅保留现有策略）。
- 权限系统、多用户系统。
- 深度历史回测平台。

### 3.3 里程碑建议
- M1：后端服务化骨架与 API 契约完成
- M2：Vue 看板页面完成并连通 WS
- M3：ALL 国内范围与策略闭环联调完成
- M4：本地部署打通，自动部署接入说明完成

## 4. 执行策略（给 Execution）

### 4.1 推荐实施顺序
1. **抽离核心服务层**
   - 从 `futures_monitor/main.py` 中抽出与 GUI 解耦的监控服务（start/stop/process tick/event emit）。
2. **新增后端接口层**
   - 建立 FastAPI app、路由、WS 广播管理。
3. **保留并改造配置机制**
   - `config.json` 继续可用；增加后端配置读写 API。
   - 前端不回传明文密码到日志。
4. **构建 Vue 前端**
   - 页面布局、状态 store、WS 客户端、表格日志渲染。
5. **ALL 国内范围收敛**
   - 在 ALL 解析后对交易所前缀过滤：CFFEX/SHFE/DCE/CZCE/INE/GFEX。
6. **联调与验收**
   - 关键流程：启动、实时更新、标记已买入、停止、异常提示。
7. **部署交付**
   - 产出前端 build、后端启动入口、环境变量模板。
   - 针对 `kline.godpenai.com` 给出反代路由建议（部署阶段套入现有自动链路）。

### 4.2 关键文件规划（Execution 预计）
- 复用：
  - `futures_monitor/market.py`
  - `futures_monitor/strategy/breakout.py`
  - `futures_monitor/strategy/state_machine.py`
  - `futures_monitor/data/storage.py`
- 新增后端（示例路径）：
  - `futures_monitor/server/app.py`
  - `futures_monitor/server/api/*.py`
  - `futures_monitor/server/services/monitor_service.py`
  - `futures_monitor/server/ws/hub.py`
- 新增前端（示例路径）：
  - `futures_monitor/web/`（Vite 工程）

### 4.3 Execution 阶段的验证门禁落点
- 后端：
  - 接口单元测试（配置、start/stop、mark-bought）
  - 事件流集成测试（至少 1 条 row/log 事件）
- 前端：
  - `npm run build`
  - 关键页面交互测试（启动/停止/日志刷新）
- 全链路：
  - 真实行情连接成功
  - ALL 模式可更新到真实合约行

## 5. 风险与应对

### 5.1 技术风险
1. **WS 连接断开导致前端不刷新**
   - 应对：前端指数退避重连 + 状态提示“重连中”。
2. **ALL 品种量大导致 UI 卡顿**
   - 应对：前端虚拟滚动/分页；后端可做节流推送。
3. **真实行情异常**
   - 应对：strict real mode 保持不回退 mock；前端明确错误展示。

### 5.2 交付风险
1. **部署环境与本地差异**
   - 应对：输出标准化产物与变量清单，部署阶段由现有自动链路适配。
2. **域名接入不一致**（`kline.godpenai.com`）
   - 应对：约定固定路径前缀：`/api`、`/ws`，避免前后端耦合环境地址。

### 5.3 回滚与兜底策略
- 保留现有 Tkinter 启动入口作为临时兜底（短期并存）。
- 新 Web 入口独立，不覆盖旧入口，便于快速回滚。
