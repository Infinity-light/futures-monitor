# 项目规划：Claude Plugin 发布器

## 1. 背景与目标

### 要解决的问题
当前团队已经明确：对 Claude Plugin / Skill 的修改，不能直接改 cache、settings 或运行态副本，而必须回到 canonical source 修改，再经版本、marketplace、云端与本地安装链路逐层同步，最后验证当前系统是否真正加载新版。

现状里虽然已有：
- `skill-creator`：偏技能创建
- `skills-updater`：偏检查更新与部分 marketplace 刷新

但仍缺少一个**面向 Claude Plugin 体系的全链路发布与生效验证 Skill**，专门负责：
- 找唯一源头
- 检查 manifest / marketplace 隐藏约束
- 同步版本与索引
- 推送云端
- 刷新本地安装
- 验证 cache 与当前系统已生效

### 目标与成功标准
首版要在 `workflow-kit` 内部落地一个新的 skill，作为 **Claude Plugin 发布器** 使用，并满足：
1. 能用于发布 `workflow-kit` 自己
2. 同时又不绑定当前单一目录结构，具备 Claude Plugin 层级的泛化能力
3. 默认支持“确认后全自动”的全链路闭环
4. 发布结束后能输出明确的成功 / 失败报告
5. 能区分：
   - 安装态已更新
   - 当前会话已确认使用新版

---

## 2. 关键决策与依据（Why）

### 2.1 放置方式
#### 候选方案对比

**方案 A：作为 workflow-kit 内部 skill（选定）**
- 优点：
  - 可以立刻自举发布 workflow-kit 自己
  - 与现有 skill 体系、hooks、CLAUDE.md、marketplace 结构天然对齐
  - 首版验证链最短，最容易快速闭环
- 风险/成本：
  - 名义上挂在 workflow-kit 内部，外部观感不如独立插件边界清晰
- 适用场景：
  - 首版要先跑通、自举验证、快速迭代

**方案 B：做成独立插件**
- 优点：
  - 边界更干净
  - 对外更像一个独立产品能力
- 风险/成本：
  - 首版就要面对“如何发布发布器自己”的 bootstrap 问题
  - 初期发布链更长，调试复杂度更高
- 适用场景：
  - 发布器已经稳定，准备独立对外提供

**方案 C：仅做脚本库，不封装 skill**
- 优点：
  - 最快
- 风险/成本：
  - 不符合 Skill-first 沉淀目标
  - 后续仍要二次封装
- 适用场景：
  - 临时内部工具，不追求长期复用

#### 最终选择
- **作为 workflow-kit 内部 skill 落地**

#### 选择理由（为什么）
- 你已经明确补充：它是个 skill，放在 workflow-kit 里。
- 这能让发布器先服务 workflow-kit 自身，形成自举闭环，再决定是否外拆。
- 这是“先跑通、再抽离”的最稳路径。

#### 不选其他方案的原因
- 独立插件过早，会把首版复杂度拉高。
- 仅脚本库违背 Skill-first 目标。

---

### 2.2 自动化程度
#### 候选方案对比

**方案 A：确认后全自动（选定）**
- 优点：
  - 满足你已经确认过的“全链路闭环”目标
  - 用户只需在高风险外部动作前确认一次，不需要手工补跑后续步骤
  - 最能减少“以为发版了其实没生效”的假阳性
- 风险/成本：
  - 需要更完整的失败分支处理
  - 需要把 push / install / verify 分层看待
- 适用场景：
  - 正式发布动作、需要结果可追溯

**方案 B：做到云端为止**
- 优点：
  - 风险更低
- 风险/成本：
  - 本地 install / 生效验证仍靠人工，闭环不完整
- 适用场景：
  - 只想做 registry 发布，不关心当前机器是否生效

**方案 C：dry-run only**
- 优点：
  - 最安全
- 风险/成本：
  - 不满足全链路目标
- 适用场景：
  - 只做规划或教学

#### 最终选择
- **确认后全自动**

#### 选择理由（为什么）
- 这是你明确选定的闭环方式。
- 只有这样，发布 Skill 才真正承担“发布与生效验证”的职责，而不是只输出一串命令。

#### 不选其他方案的原因
- 到云端为止仍存在本地未生效的关键不确定性。
- dry-run 违背本轮目标。

---

### 2.3 拓扑发现方式
#### 候选方案对比

**方案 A：自动发现 + 显式覆盖（选定）**
- 优点：
  - 泛化性与易用性最平衡
  - 既能适配当前 cytopia-workflow，也能适应未来路径变化
  - 当自动发现失败时，可退回用户显式指定
- 风险/成本：
  - 需要定义清楚发现顺序和冲突处理规则
- 适用场景：
  - 首版通用化 Skill

**方案 B：全部显式传入**
- 优点：
  - 最少依赖约定
- 风险/成本：
  - 使用成本高
  - 不利于日常自举发布
- 适用场景：
  - 环境极度异构

**方案 C：强依赖当前目录约定**
- 优点：
  - 落地最快
- 风险/成本：
  - 一旦仓库路径变化就失效
  - 与你的泛化要求冲突
- 适用场景：
  - 纯内部固定环境

#### 最终选择
- **自动发现 + 显式覆盖**

#### 选择理由（为什么）
- 能保留当前工作流的效率，同时不把 skill 绑死在当前目录结构上。

#### 不选其他方案的原因
- 全显式太重。
- 强依赖约定不满足泛化要求。

---

### 2.4 抽象层级
#### 候选方案对比

**方案 A：仅 workflow-kit 发布器**
- 优点：最快
- 缺点：复用性差

**方案 B：Claude Plugin 发布器（选定）**
- 优点：
  - 贴合稳定结构：source / manifest / marketplace / cache / installed state
  - 泛化性足够，且不会过度抽象
- 缺点：
  - 需要明确不变量 / 变量

**方案 C：任意扩展包发布器**
- 优点：理论最泛
- 缺点：
  - 首版过重
  - 规则漂移大

#### 最终选择
- **Claude Plugin 发布器**

#### 选择理由（为什么）
- 当前问题空间本来就属于 Claude Plugin 结构，抽象到这一层最稳。

#### 不选其他方案的原因
- workflow-kit 专用太窄。
- 任意扩展发布器过宽。

---

### 2.5 验证深度
#### 候选方案对比

**方案 A：只看 install 成功提示**
- 缺点：假阳性高

**方案 B：installed_plugins.json + cache 双层验证**
- 优点：安装态证据强
- 缺点：仍可能看不出目标改动是否真正进入运行文件

**方案 C：installed_plugins.json + cache + 内容锚点三层验证（选定）**
- 优点：
  - 能验证版本记录、安装结果和实际内容都已更新
  - 对 Skill 文本、规则文案等改动尤其有效
- 缺点：
  - 需要用户或流程提供验证锚点

#### 最终选择
- **三层验证**

#### 选择理由（为什么）
- 只有这样，才能减少“版本号更新了但目标变更没进去”的风险。

#### 不选其他方案的原因
- install 成功提示不足以证明生效。
- 双层验证仍缺对目标内容的确认。

## 3. 实施范围与阶段划分

### 本期范围
首版只覆盖：
- Claude Plugin 结构
- workflow-kit 内部新增一个 skill
- canonical source → version/index → remote → local install → verify 的全链路
- 发布报告与失败阻断规则

### 非本期范围
- 不覆盖 npx skills / skills.sh 的完整生态
- 不覆盖 GitHub Release / Tag / 多分支发布策略
- 不覆盖“任意扩展包”的超通用抽象
- 不自动替用户跳过高风险外部动作确认

### 里程碑建议
1. **里程碑 1：Skill 设计落地**
   - 新 skill 的 SKILL.md 结构
   - references / scripts 的边界
   - 发现逻辑与输入输出定义
2. **里程碑 2：workflow-kit 自举发布闭环**
   - 用该 skill 发布 workflow-kit 自己
   - 完成 Discovery 改动的真实发版与本地生效验证
3. **里程碑 3：泛化能力验证**
   - 在第二个 Claude Plugin 上复用一遍流程
   - 检查哪些路径假设要进一步抽象

## 4. 执行策略（给 Execution）

### 推荐实施顺序
1. 在 `workflow-kit` 内新增发布 skill 的目录与主体 SKILL
2. 把规则拆成：
   - SKILL.md：总流程、决策规则、风险边界
   - references：manifest 约束、状态文件结构、验证规则
   - scripts：检测/验证/辅助脚本（若必要）
3. 先实现“拓扑发现 + 发布前检查”
4. 再实现“版本/索引同步”
5. 再实现“云端发布 + 本地刷新”
6. 最后实现“三层验证 + 发布报告”
7. 完成后，用这个 skill 真正发布本次 Discovery 改动

### 关键依赖与风险前置
- 需要先明确当前 workflow-kit 的 canonical source、remote、marketplace 名称
- 需要发布前检查 `plugin.json` 隐藏约束
- 需要明确哪些动作属于高风险外部操作，执行前需用户确认
- 需要为“内容锚点验证”定义输入规范

### Execution 阶段的验证与测试门禁
Execution 至少要验证：
1. 新 skill 能正确定位：
   - source root
   - plugin.json
   - marketplace.json
   - installed_plugins.json
   - cache installPath
2. 能区分并阻断失败场景：
   - 源头不唯一
   - manifest 不合法
   - 版本未同步
   - remote 未成功
   - install 未刷新
3. 能输出结构化发布报告
4. 能在 workflow-kit 自身上跑通一次真实闭环

---

## 5. 风险与应对

### 技术风险
#### 风险 1：manifest 校验规则与公开文档不一致
- 应对：把隐藏约束检查内置到 Skill 工作流里，作为发布前阻断项。

#### 风险 2：本地 install 成功但运行态未实际更新
- 应对：采用 installed_plugins.json + cache + 内容锚点三层验证。

#### 风险 3：自动发现误判源头
- 应对：采用“自动发现 + 显式覆盖”，一旦发现冲突立即要求用户确认源头。

### 交付风险
#### 风险 4：首版抽象过度，导致落地变慢
- 应对：限制在 Claude Plugin 结构层，不上升到任意扩展包生态。

#### 风险 5：放成独立插件导致 bootstrap 过重
- 应对：首版先放在 workflow-kit 内部 skill，先完成自举。

### 回滚与兜底策略
- 若 manifest 不合法：停止发布，不继续向下游传播
- 若云端发布失败：保留本地修改与版本同步结果，但标记“未完成发布”
- 若本地刷新失败：标记“安装态未生效”
- 若内容锚点未命中：标记“运行态疑似未生效”，不得宣称完成
