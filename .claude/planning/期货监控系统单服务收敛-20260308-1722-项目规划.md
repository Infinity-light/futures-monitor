# 项目规划：期货监控系统单服务收敛

## 1. 背景与目标

### 1.1 要解决的问题
当前项目已经完成了 FastAPI 后端与 Vue 前端的 Web 化基础建设，但部署形态仍延续“backend/frontend 双服务”思路。经对 deploy-setup 源码与模板的核查，当前工具只稳定支持单服务模板，不兼容当前双服务交付结构。

因此，本次规划的目标不是推翻现有技术路线，而是把已存在的 FastAPI + Vue 方案收敛成 deploy-setup 可直接识别和自动部署的单服务架构。

### 1.2 目标与成功标准
- 保留 FastAPI 提供 `/api/*` 与 `/ws/events` 的能力。
- 保留 Vue 前端工程，但 build 后并入后端运行镜像。
- 对外只保留：一个服务、一个端口、一个 Dockerfile、一个 compose service、一个 deploy-setup APP_PORT。
- 通过 deploy-setup 自动生成与部署产物，不再与模板假设冲突。
- 最终能跑通 execution → deploy → verification。

成功标准：
1. 浏览器打开域名即可看到 Vue 页面。
2. `/api/health` 正常。
3. `/ws/events` 正常。
4. deploy-setup 能安全识别并生成单服务产物。
5. 自动部署完成后，页面、API、WS 都由同一服务对外提供。

---

## 2. 关键决策与依据（Why）

### 2.1 部署形态：单服务收敛

**候选方案**
1) 保持 backend/frontend 双服务
2) FastAPI 统一提供 API + WS + Vue 静态资源（单服务）
3) 前端外部托管（CDN/Nginx），后端单独部署

**最终选择**：FastAPI 单服务收敛（用户已确认）

**选择理由**
- deploy-setup 当前模板明确假设单服务、单端口、单 Dockerfile、单 compose service。
- 当前工具无法安全表达 backend/frontend 双 service，因此若不收敛会持续与部署工具冲突。
- 单服务收敛不会放弃 Vue，也不会放弃 FastAPI，只是改变交付与对外暴露方式。

**不选其他方案原因**
- 保持双服务会继续与 deploy-setup 的生成器、compose 模板、workflow 模板冲突。
- 外部静态托管虽然技术上可行，但会重新引入额外边界，不利于当前“标准化自动部署”目标。

---

### 2.2 Vue 静态资源并入方式

**候选方案**
1) 后端镜像内置 Vue build 产物
2) 运行时挂载 dist 目录
3) 保留外部静态托管

**最终选择**：后端镜像内置（用户已确认）

**选择理由**
- 最符合 deploy-setup 的单镜像模型。
- 构建产物与后端代码一起进入镜像，部署最稳定、最少依赖外部目录状态。
- 对最终交付者来说，域名下页面/API/WS 是一致的，不存在“页面资源还得单独找”的问题。

**不选其他方案原因**
- 运行时挂载 dist 目录会增加部署时文件存在性与目录同步问题。
- 外部静态托管会重新引入第二条交付链路。

---

### 2.3 前端路由兼容策略

**候选方案**
1) History 路由 + 后端 index.html 回退
2) Hash 路由
3) 仅保留单页根路由

**最终选择**：History + index 回退（用户已确认）

**选择理由**
- URL 更标准，专业感和用户体验更好。
- 与现有 Vue 工程方向兼容，不需要为了部署妥协成 hash URL。
- 在单服务架构中，由 FastAPI 实现 `/api`、`/ws` 排除 + `index.html` 回退，是标准成熟做法。

**不选其他方案原因**
- Hash 路由虽然省事，但会降低标准 Web 控制台体验。
- 仅单页根路由会过度限制后续扩展。

---

### 2.4 运行入口

**候选方案**
1) FastAPI 直接启动（uvicorn）
2) Nginx 前置 + FastAPI 后置
3) 更换为其他 Python 网关/服务容器

**最终选择**：FastAPI 直接启动（用户已确认）

**选择理由**
- 当前后端已经有 `futures_monitor/server/app.py`，保留为唯一入口最自然。
- API、WS、静态页面都由同一应用承接，结构最清晰。
- deploy-setup 单服务模板只需要理解一个 `startCmd`，与此模型完全契合。

**不选其他方案原因**
- Nginx 前置虽然常见，但在当前目标下会重新增加运行时层次，接近回到双边界部署。
- 更换入口会带来不必要的额外改造。

---

### 2.5 构建策略

**候选方案**
1) 单 Dockerfile 多阶段构建（先 build Vue，再打包 FastAPI）
2) 两个 Dockerfile 最后拼接
3) 本地先 build 再复制进容器

**最终选择**：单 Dockerfile 多阶段构建

**选择理由**
- 单 Dockerfile 是 deploy-setup 兼容的关键条件之一。
- 多阶段构建可以同时满足：前端独立 build、运行镜像内置静态产物、最终单镜像交付。
- 可控且符合现代容器实践。

**不选其他方案原因**
- 两个 Dockerfile 会保留双服务痕迹，继续与 deploy-setup 冲突。
- 本地先 build 再复制会让 CI/CD 过程变脆弱，不利于自动部署可重复性。

---

## 3. 实施范围与阶段划分

### 3.1 本期范围
1. 单服务架构重构：
   - FastAPI 统一提供静态资源、API、WS。
2. Vue 构建产物并入：
   - 保留 `futures_monitor/web` 工程。
   - build 产物纳入后端镜像。
3. 路由兼容：
   - `/api/*` 与 `/ws/*` 保留。
   - 前端 history 路由回退到 `index.html`。
4. 部署收敛：
   - 单 Dockerfile
   - 单 compose service
   - 单 APP_PORT
   - deploy-setup 兼容配置与部署
5. 验证闭环：
   - 页面访问、API、WS、deploy-setup 识别、自动部署结果验证。

### 3.2 非本期范围
- 自动下单能力。
- 新策略引擎开发。
- 多租户或权限系统。
- 再引入独立前端容器或外部静态托管体系。

### 3.3 里程碑建议
- M1：后端单服务入口重构完成
- M2：Vue 静态资源收敛与路由回退完成
- M3：单镜像/单端口/单 compose service 完成
- M4：deploy-setup 接入与自动部署完成
- M5：Verification 验收完成

---

## 4. 执行策略（给 Execution）

### 4.1 推荐实施顺序
1. **清理部署目标模型**
   - 以 `futures_monitor/server/app.py` 为唯一运行入口。
   - 明确未来只保留一个对外端口。

2. **实现静态资源托管**
   - FastAPI 挂载 Vue build 产物目录。
   - 为未知前端路径回退到 `index.html`。
   - 严格排除 `/api/*` 和 `/ws/*`。

3. **收敛前端工程交付**
   - 保留 `futures_monitor/web` 开发方式。
   - 将其 build 步骤纳入后端镜像构建流程。

4. **收敛容器与部署文件**
   - 删除或停用 backend/frontend 双服务部署假设。
   - 统一为 deploy-setup 可识别的单 Dockerfile / 单 compose service / 单 APP_PORT。

5. **接入 deploy-setup**
   - 用非交互 JSON 配置推进 `deploy-setup init/all`。
   - 使用已确认服务器与域名。

6. **验证与回归**
   - 页面能打开
   - `/api/health` 正常
   - `/ws/events` 正常
   - 自动部署成功

### 4.2 关键文件规划（Execution 预计）
- 重点改造：
  - `futures_monitor/server/app.py`
  - `futures_monitor/server/...`
  - `futures_monitor/web/...`
  - `futures_monitor/Dockerfile`
  - `docker-compose.yml`
  - `.github/workflows/deploy.yml`
- deploy-setup 对接文件：
  - `.deploy-setup.input.json`（或同类输入配置）
  - `deploy-config.json`
  - `.deploy-setup-cache.json`
  - 由 deploy-setup 生成/更新的部署文件

### 4.3 Execution 阶段的验证门禁落点
- Python 验证：
  - `python -X utf8 .claude/validate.py`
  - 单元/集成测试全量通过
- 前端验证：
  - `npm run build`
- 单服务验证：
  - 浏览器首页返回 Vue 页面
  - `/api/health` 返回 200
  - `/ws/events` 可握手连接
- 部署验证：
  - deploy-setup 可完成 init/all
  - GitHub Actions 成功
  - 域名下页面、API、WS 可用

## 5. 风险与应对

### 5.1 技术风险
1. **静态资源托管与 API/WS 路由冲突**
   - 应对：路由优先级明确，先匹配 `/api`、`/ws`，再处理静态和 index 回退。

2. **Vue history 路由误返回 404**
   - 应对：专门实现前端路由 fallback 逻辑，并加集成验证。

3. **单镜像构建过程变长**
   - 应对：采用多阶段构建，分别缓存 Node/Python 依赖层。

4. **ALL 模式与 WebSocket 大流量压力**
   - 应对：保持国内期货范围过滤；必要时在后端增加节流。

### 5.2 交付风险
1. **旧双服务部署文件残留干扰**
   - 应对：Execution 阶段明确收敛基线，只保留单服务部署产物。

2. **deploy-setup 重新覆盖文件造成不一致**
   - 应对：以 deploy-setup 兼容结构为先，再执行 init/all，避免再次对冲。

3. **自动部署链路失败**
   - 应对：先本地跑通单服务，再进行 deploy；保留日志与健康检查证据。

### 5.3 回滚与兜底策略
- 保留当前 Web 代码资产，不回退 Tkinter。
- 若 deploy-setup 接入失败，回滚到“本地已验证通过的单服务结构”，继续排查部署模板，而不是回到双服务。
