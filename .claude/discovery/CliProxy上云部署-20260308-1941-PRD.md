# PRD: CliProxy 上云部署

## 一、需求理解（为什么）

### 1. 目标
用户希望把当前在本地运行的 CliProxy 服务部署到云端，供多人稳定访问和使用。现阶段目标不是重构产品，也不是改造认证机制，而是在尽量少改动现有运行形态的前提下，快速形成一个可落地、可维护、可控风险的最小可行部署方案。

### 2. 已知事实
- 当前服务以 EXE 形式启动。
- 当前监听端口为 `8317`。
- 配置文件位于 EXE 对应目录下。
- 用户已有域名，也具备购买云服务器的条件。
- 目标是多人共用云上的 CliProxy 服务。
- 不包括注册机，不讨论任何绕过验证相关内容。

### 3. 用户真实诉求拆解
从当前上下文看，用户的核心诉求不是“做一个完整云原生平台”，而是：
1. 让现有本地 EXE 服务尽快迁移到云端；
2. 让多人能够通过域名稳定接入；
3. 尽量减少对现有运行方式、配置方式、部署方式的改造；
4. 增加最基本的公网暴露安全措施，避免服务和管理面直接裸奔；
5. 用最少组件完成上线，先跑通，再决定是否做后续扩展。

### 4. 交付物清单（最小单元）
本次 discovery 的交付物应聚焦以下最小可验证单元：
1. 云上部署形态建议；
2. 操作系统选型建议；
3. 单机部署拓扑建议；
4. 反向代理与 HTTPS 接入建议；
5. 管理面访问控制建议；
6. 最小步骤清单；
7. 用户接入方式说明；
8. 验收标准。

### 5. 边界条件

#### 范围内
- 将现有 EXE 形态部署到云服务器；
- 使用域名对外提供访问；
- 增加前置反向代理与 HTTPS；
- 给管理面增加最基本访问限制；
- 形成单机最小可行上线方案。

#### 范围外
- 不讨论注册机；
- 不讨论绕过授权、绕过验证、破解、逆向等内容；
- 不做微服务拆分；
- 不做 Kubernetes、容器编排、高可用集群等过度设计；
- 不要求本次就实现自动化运维平台或复杂监控体系；
- 不要求改写现有 EXE 为 Linux 原生服务。

## 二、调研结论（是什么）

### 1. 需求本质
这不是一个“重新设计 CliProxy 产品架构”的问题，而是一个“将既有 Windows EXE 服务以最低改造成本搬到公网环境并安全暴露”的问题。

因此，方案判断标准应以以下优先级排序：
1. 能否直接承载现有 EXE；
2. 是否最少改造；
3. 是否最容易上线与维护；
4. 是否能满足多人接入；
5. 是否具备基础安全能力。

### 2. 最小模块拆解

#### 模块 A：运行环境承载
- A1：选择云服务器操作系统
- A2：部署 EXE 主程序
- A3：管理配置文件与运行目录

#### 模块 B：公网接入
- B1：域名解析到云服务器
- B2：反向代理转发到本机 `8317`
- B3：配置 HTTPS 证书

#### 模块 C：安全控制
- C1：限制管理面公网暴露
- C2：主机防火墙最小开放
- C3：基础访问审计与运维入口控制

#### 模块 D：使用与运维
- D1：用户接入方式定义
- D2：服务启动与重启方式
- D3：基础上线验证方法

### 3. 方案对比

| 方案 | 描述 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| 方案一：Windows 云服务器 + EXE 直跑 + IIS/Nginx 反代 | 直接在 Windows 云主机上运行现有 EXE，并由反向代理对外提供 HTTPS | 改造最少、最贴合现状、最快上线 | 运维规范需靠人工补齐 | 推荐，作为最小可行方案 |
| 方案二：Linux 云服务器 + Wine/兼容层运行 EXE | 保持 EXE 不变，但在 Linux 上通过兼容层运行 | 理论上成本可能更低 | 兼容性、稳定性、排障复杂度高 | 不推荐作为首选 |
| 方案三：重写/改造为 Linux 服务或容器化服务 | 从根本上改造成适合云环境的形态 | 长期架构更标准 | 成本高、周期长、偏离当前目标 | 当前阶段不推荐 |

### 4. 推荐结论

#### 4.1 推荐优先使用 Windows 云服务器
原因很明确：
- 当前程序启动方式就是 EXE；
- 配置文件与 EXE 同目录，说明当前运行形态较偏向传统 Windows 应用部署；
- 直接迁移到 Windows 云主机可以最大限度复用现有运行方式；
- 避免 Linux 兼容层带来的不确定性；
- 对于“先上云给多人用”这个目标，Windows 单机是最短路径。

#### 4.2 推荐单机部署即可
当前目标是“最小可行方案”，不是高可用建设。单机部署已经足以满足：
- 一个公网入口；
- 一份程序目录；
- 一份配置文件；
- 一个反向代理；
- 一套 HTTPS；
- 多人共享访问。

在未验证真实使用规模前，不建议上来就做双机、负载均衡、数据库拆分、容灾切换等设计。

#### 4.3 必须有前置反向代理和 HTTPS
不建议让外部用户直接访问 `8317` 端口。推荐结构为：
- 公网用户访问域名；
- 域名进入前置反向代理（IIS + ARR 或 Windows 版 Nginx 均可）；
- 反向代理将请求转发到本机 `127.0.0.1:8317`；
- 由反向代理统一处理 TLS/HTTPS、域名接入、可选访问控制。

这样做的价值：
- 避免业务端口直接暴露公网；
- 简化证书部署；
- 便于后续增加白名单、Basic Auth、来源限制等能力；
- 用户接入体验更统一，直接用域名即可。

#### 4.4 管理面不要直接裸暴露公网
如果 CliProxy 存在管理面、控制台、后台接口，不应直接无保护暴露在公网。最低要求应包括以下之一，最好组合使用：
- 仅允许特定 IP 访问；
- 在反向代理层增加认证；
- 将管理面放在单独子路径或子域名并加限制；
- Windows 防火墙只允许必要来源；
- 远程运维入口（如 RDP）限制来源 IP 和强密码。

最小原则是：业务访问可以公网可达，但管理入口至少要有访问限制，不能裸放。

### 5. 面向最小可行方案的部署拓扑

```text
使用者浏览器 / 客户端
        ↓
     域名解析
        ↓
Windows 云服务器
        ↓
反向代理（HTTPS 终止）
        ↓
127.0.0.1:8317
        ↓
CliProxy EXE
        ↓
EXE 同目录配置文件
```

### 6. 深度/广度/前瞻性结论

#### 6.1 深度结论
- 现有程序是 EXE，说明当前最稳妥承载环境是 Windows；
- 配置文件与 EXE 同目录，说明部署单元天然适合“整目录上传 + 原位运行”；
- 监听端口已知为 `8317`，反向代理配置可直接围绕该端口建立；
- 若采用“本机监听 + 反代暴露”模式，可把服务进程与公网入口解耦，提升安全性和可维护性。

#### 6.2 广度结论
主流可选路径并不少，但结合当前信息，真正合适的只有“Windows 单机 + 反向代理 + HTTPS”这一条。它在实现成本、排障成本、认知负担上都最优。

#### 6.3 前瞻性结论
长期看，若未来出现以下情况，可以再升级架构：
- 使用人数明显增长；
- 需要灰度发布或自动扩缩；
- 需要高可用；
- 需要更标准化运维；
- 需要容器化和自动化交付。

但在当前阶段，最先进的方案不等于最合适的方案。最合适的是最小可行、快速可上线、低改造成本的方案。

## 三、实现路径（怎么办）

### 1. 推荐实现方案
采用“Windows 云服务器单机部署 + 前置反向代理 + HTTPS + 管理面访问限制”的最小可行方案。

### 2. 用户动线

#### 2.1 部署者动线
1. 购买一台 Windows 云服务器；
2. 将域名解析到该云服务器公网 IP；
3. 上传现有 CliProxy EXE 目录到服务器；
4. 确认配置文件位于 EXE 同目录并按云环境调整；
5. 在服务器上启动 EXE，使其监听本机 `8317`；
6. 安装并配置前置反向代理，将域名请求转发到 `127.0.0.1:8317`；
7. 配置 HTTPS 证书；
8. 对管理面增加访问限制；
9. 开放必要端口并关闭不必要暴露；
10. 使用域名从外部验证访问；
11. 将接入地址发给使用者。

#### 2.2 使用者动线
1. 获取统一接入域名；
2. 通过 HTTPS 域名访问云上 CliProxy；
3. 按既定客户端或调用方式接入服务；
4. 无需关心服务器 IP、EXE 目录、内部端口 `8317`；
5. 如需管理操作，则通过受限入口访问，不直接走开放公网管理面。

### 3. 系统架构

```text
[公网用户]
   ↓ HTTPS
[域名]
   ↓
[Windows 云服务器]
   ├─ [反向代理：IIS/Windows Nginx]
   │      ↓ 转发
   │   [127.0.0.1:8317]
   │
   ├─ [CliProxy EXE]
   │      ↓
   │   [同目录配置文件]
   │
   └─ [Windows 防火墙 / 访问限制 / 运维入口控制]
```

### 4. 数据流与关键接口
- 用户外部访问入口：`https://你的域名`
- 反向代理内部转发目标：`http://127.0.0.1:8317`
- 程序配置读取位置：EXE 所在目录下的配置文件
- 管理入口：应通过单独路径、单独域名或代理规则加访问限制

### 5. 最小步骤清单
这是本次最推荐的落地顺序：
1. 购买 Windows 云服务器；
2. 绑定公网 IP 并完成域名解析；
3. 上传现有 EXE 及其同目录配置文件；
4. 在服务器本机确认 EXE 可以正常启动；
5. 确认本机 `8317` 可用；
6. 安装并配置反向代理；
7. 让外部只通过域名 + HTTPS 访问，不直接暴露 `8317`；
8. 给管理面配置访问限制；
9. 配置 Windows 防火墙，仅开放必要端口（如 80/443，远程管理端口按需限源）；
10. 从公网环境完成最终验证；
11. 将接入文档化为最小说明发给使用者。

### 6. 用户接入方式
推荐统一对外只提供以下信息：
- 一个域名地址，例如：`https://cliproxy.example.com`
- 一份最简接入说明（访问地址、是否需要认证、注意事项）
- 如有管理入口，应单独说明其访问限制，不与普通用户入口混用

不建议直接把以下信息暴露给普通使用者：
- 服务器公网 IP；
- 服务内部端口 `8317`；
- 服务器远程桌面入口；
- 管理后台裸地址。

### 7. 风险与控制

#### 风险 1：直接暴露 8317
- 问题：业务端口直接裸露，安全面扩大；
- 控制：仅让反向代理对外开放，`8317` 仅本机可达或仅内网可达。

#### 风险 2：管理面裸暴露
- 问题：后台接口被探测、暴力尝试或误操作；
- 控制：至少做 IP 限制或代理层认证。

#### 风险 3：Windows 云主机成为单点
- 问题：单机故障即服务中断；
- 控制：当前阶段接受此现实，先保证可用；后续如验证有必要，再做高可用升级。

#### 风险 4：配置文件与程序目录耦合
- 问题：升级覆盖时容易误改配置；
- 控制：保留目录备份，升级前备份配置文件，形成最小变更流程。

## 四、验收标准

### 1. 基础可用性
- 云服务器上可成功启动现有 CliProxy EXE；
- 服务可在服务器本机正常监听 `8317`；
- 配置文件可从 EXE 同目录正常读取；
- 服务重启后可恢复运行。

### 2. 对外访问能力
- 域名已正确解析到云服务器；
- 用户可通过 HTTPS 域名访问服务；
- 外部访问不需要直接连接 `8317`；
- 多名用户可共享同一云上服务入口。

### 3. 安全最小要求
- 已配置前置反向代理；
- 已启用 HTTPS；
- 管理面未裸暴露公网；
- 至少具备一种访问限制手段（IP 白名单、代理认证、独立受限入口等）；
- Windows 防火墙仅开放必要端口。

### 4. 运维最小要求
- 部署步骤可重复；
- 服务器上有明确的 EXE 目录与配置文件位置；
- 出现问题时可以定位到域名、反向代理、EXE 进程、配置文件四个层次；
- 使用者拿到域名后即可接入，无需了解服务器内部实现。

### 5. 最终验收结论
满足以下四点即可视为本次“CliProxy 上云部署”最小可行方案验收通过：
1. EXE 已在 Windows 云服务器稳定运行；
2. 域名 + HTTPS 已可用；
3. 单机部署已可供多人访问；
4. 管理面未裸暴露公网并已具备基础访问限制。
