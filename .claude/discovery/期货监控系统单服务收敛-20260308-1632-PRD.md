# PRD: 期货监控系统单服务收敛

## 一、需求理解（为什么）

### 1.1 背景
当前项目已经完成了从 Tkinter 桌面界面向 Web 架构迁移的第一阶段探索：
- 后端已具备 FastAPI 服务入口，可提供 API 与 WebSocket；
- 前端已具备独立 Vue 工程，可单独开发与构建；
- 旧方案默认采用 backend/frontend 双服务部署思路。

但最新 deploy-setup 研究结论已经明确：**当前 deploy-setup 是单服务模板，不兼容 backend/frontend 双服务部署结构；但兼容“FastAPI 同时提供 API + WS + Vue 静态资源”的单服务模型。**

因此，本次 Discovery 的核心目标，不是回退技术路线，而是将已经确定的“FastAPI + Vue”Web 化方向，进一步收敛为 **deploy-setup 可直接识别、可自动部署、可单端口暴露** 的单服务交付形态。

### 1.2 目标
本期仅完成 Discovery PRD，明确新的部署与交付方向：
- 将当前双服务部署结构收敛为单服务架构；
- 保留 FastAPI 后端的 API 与 WebSocket 能力；
- 保留 Vue 前端工程，但改为 build 后由 FastAPI/同一容器对外统一提供；
- 使项目输出满足 deploy-setup 的自动部署要求。

### 1.3 为什么必须从双服务收敛到单服务
#### 1) 部署兼容性要求
现有 deploy-setup 只能稳定处理单服务模板，无法直接识别“一个后端服务 + 一个前端服务”的双容器/双 service 结构。若继续沿用双服务：
- 自动生成产物时会出现识别歧义；
- 环境变量注入、端口映射、容器编排会变复杂；
- 自动部署链路需要额外定制，不符合当前模板假设。

#### 2) 自动部署要求
当前目标不是“本地能跑即可”，而是要进入 deploy-setup 兼容体系，保证：
- 一个 APP_PORT；
- 一个 Dockerfile；
- 一个 compose service；
- 一个对外服务入口；
- 自动部署工具能安全生成、渲染、发布产物。

双服务架构天然会引入：
- 两个端口；
- 前后端服务发现与反代配置；
- WebSocket 转发配置；
- 额外的部署模板复杂度。

这些都与 deploy-setup 当前单服务模板不匹配。

#### 3) 交付一致性要求
对最终使用者来说，期望是：
- 打开域名即可看到完整页面；
- 同域名下 API 正常；
- 同域名下 WebSocket 正常；
- 不关心前端资源来自哪个独立服务。

将 Vue build 产物纳入 FastAPI 同一服务统一对外暴露，可以减少：
- 跨域问题；
- 反向代理复杂度；
- 多服务联调成本；
- 自动部署失败面。

### 1.4 本次需求的精确定义
本次“单服务收敛”的精确定义如下：

1. **FastAPI 仍然保留 API 能力**
   - `/api/*` 路由继续存在。
   - 例如 `/api/health`、监控配置接口、启动/停止监控接口等仍由 FastAPI 提供。

2. **FastAPI 仍然保留 WebSocket 能力**
   - `/ws/events` 继续存在。
   - 实时日志、连接状态、监控行状态仍由同一个 FastAPI 进程推送。

3. **Vue 前端仍然保留**
   - 不删除 Vue 工程；
   - 不改回 Tkinter；
   - 不放弃 Web 前端。

4. **Vue 的交付方式改变**
   - Vue 仍独立开发、独立 build；
   - build 后产生的静态资源并入后端镜像；
   - 由 FastAPI/同一容器统一对外提供页面与静态资源。

5. **对外只保留一个服务边界**
   - 一个服务；
   - 一个端口；
   - 一个 Dockerfile；
   - 一个 compose service；
   - 一个 deploy-setup APP_PORT。

### 1.5 边界
#### 明确在范围内
- 调整部署形态与交付模型；
- 规划静态资源托管方案；
- 规划前端路由回退；
- 规划单镜像构建与 deploy-setup 兼容形态；
- 保留现有 API/WS 契约边界。

#### 明确不在范围内
- 不是回退 Tkinter；
- 不是放弃 Vue；
- 不是重写监控策略逻辑；
- 不是本次直接做代码实现；
- 不是引入新的多服务网关系统。

### 1.6 交付物清单（最小可验证单元）
1. 单服务部署目标定义
2. 模块级收敛拆解
3. 用户动线说明
4. 系统架构说明
5. 数据流与路由边界说明
6. deploy-setup 兼容验收标准

### 2.3 三重维度尽调

#### 模块 A：单服务入口统一

**深度**
- 当前 FastAPI 已经是后端主入口，且已经包含 `/api/health` 与 `/ws/events`。
- 因此最合理的单服务收敛方式，不是增加新的网关层，而是继续以 FastAPI 作为唯一对外应用入口。
- 这意味着：
  - API 请求继续进入 FastAPI；
  - WebSocket 请求继续进入 FastAPI；
  - 浏览器访问根路径 `/` 时，由 FastAPI 返回 Vue 构建后的静态页面。

**广度（方案对比）**

| 方案 | 描述 | 优点 | 缺点 | 适配度 |
|---|---|---|---|---|
| FastAPI 同时提供 API + WS + 静态资源 | 单进程统一入口 | 与现有后端最一致，部署最简单 | 静态资源能力不如专用网关灵活 | 高 |
| Nginx 提供静态资源，FastAPI 提供 API/WS | 双服务协同 | 业界常见 | 与 deploy-setup 单服务模板不兼容 | 低 |
| 前端 Node 服务 + 后端 FastAPI | 双服务运行时 | 前端工程独立性强 | 端口与 service 翻倍 | 低 |

**前瞻性**
- 对于中小型内部系统、控制台、监控看板，FastAPI 直接托管 SPA build 产物是 2024-2026 年仍然常见且可维护的路径。
- 尤其在有“自动部署模板必须单服务”的前提下，这种模式比标准双服务更符合工程现实。

**结论**
- 推荐方案：**FastAPI 作为唯一对外进程入口，同时提供 API、WS、静态页面。**

---

#### 模块 B：前端静态资源并入

**深度**
- Vue 前端仍然保持独立工程开发模式；
- 但交付时执行 `npm run build`，生成 `dist/` 或同类构建产物；
- 后端镜像构建时，将该构建产物复制进后端可访问目录；
- FastAPI 将其作为静态资源目录挂载。

这满足两个目标：
1. 保留 Vue 工程的现代前端开发体验；
2. 对外只呈现一个服务。

**广度（方案对比）**

| 方案 | 描述 | 优点 | 缺点 | 适配度 |
|---|---|---|---|---|
| Vue build 后并入后端镜像 | 构建产物与后端同镜像发布 | 符合单服务、单镜像要求 | 构建流程需串联前后端 | 高 |
| 前端 dist 独立上传到对象存储/CDN | 静态资源外置 | 可缓存、可扩展 | 形成额外外部依赖，不是单服务 | 中低 |
| 前端独立容器运行 | 双服务 | 前后端职责清晰 | 不兼容 deploy-setup | 低 |

**前瞻性**
- 对于需要快速自动部署、单租户或轻量控制台场景，前端 build 产物并入后端镜像仍是稳定主流做法。
- 该模式在未来若要升级为 CDN/Nginx 前置，也仍可平滑迁移，因此不是死路。

**结论**
- 推荐方案：**Vue 工程保留，构建产物并入 FastAPI 镜像，由同一容器统一托管。**

---

#### 模块 C：浏览器路由兼容

**深度**
- Vue SPA 若使用 history 路由，浏览器直接访问非根路径页面时，后端必须把未知前端路由回退到 `index.html`；
- 但 `/api/*` 与 `/ws/*` 不能被错误回退到 `index.html`；
- 因此必须明确：
  - API 路由优先；
  - WS 路由优先；
  - 静态文件命中则直接返回；
  - 其他前端路径回退到 `index.html`。

**广度（方案对比）**

| 方案 | 优点 | 缺点 | 适配度 |
|---|---|---|---|
| 后端路由回退到 index.html | 兼容 SPA history 路由 | 需谨慎排除 /api 与 /ws | 高 |
| Vue 改 hash 路由 | 无需回退配置 | URL 形态较差，弱化标准 Web 体验 | 中 |
| 独立 Nginx 做路由回退 | 逻辑清晰 | 又回到双服务部署 | 低 |

**前瞻性**
- 现代 SPA 在单服务模式下，后端处理 index 回退是成熟常规方案。

**结论**
- 推荐方案：**在 FastAPI 中实现静态资源 + 前端路由回退，并明确排除 `/api` 与 `/ws`。**

---

#### 模块 D：镜像与部署收敛

**深度**
单服务在本项目中的部署定义必须落到可执行的产物结构：
- 一个 Dockerfile：负责前端构建与后端运行；
- 一个运行容器：同时承载静态资源、API、WS；
- 一个对外端口：由 `APP_PORT` 控制；
- 一个 compose service：deploy-setup 只需识别这一服务；
- 一个启动入口：例如 uvicorn/gunicorn + FastAPI app。

**广度（方案对比）**

| 方案 | 描述 | 优点 | 缺点 | 适配度 |
|---|---|---|---|---|
| 单 Dockerfile + 单 service + 单 APP_PORT | deploy-setup 友好 | 模板最简单 | 镜像职责集中 | 高 |
| 双 Dockerfile + compose 双 service | 前后端独立 | 扩展灵活 | 当前模板无法兼容 | 低 |
| 多阶段外部构建 + 运行时拼接 | 可减小镜像 | 工程复杂度更高 | 中 |

**前瞻性**
- 单镜像、多阶段构建是 2024-2026 年常规实践，既能保持单服务部署，又能控制镜像层次和产物复制流程。

**结论**
- 推荐方案：**单 Dockerfile、单 compose service、单 APP_PORT，作为 deploy-setup 唯一识别单元。**

---

#### 模块 E：验证闭环

**深度**
单服务收敛不是“能启动即可”，而是必须覆盖三类对外能力：
1. 页面访问能力；
2. API 健康检查能力；
3. WebSocket 实时连接能力。

同时还要验证 deploy-setup 对结构的识别安全性，避免出现：
- 误识别成双服务；
- 自动生成错误端口映射；
- 漏掉前端静态产物。

**广度（方案对比）**

| 验证方式 | 覆盖范围 | 价值 |
|---|---|---|
| 浏览器手工验收 | 页面与用户动线 | 直接确认对外效果 |
| `/api/health` 接口验收 | API 层 | 确认服务活性 |
| `/ws/events` 连接验收 | 实时通道 | 确认 WS 保留 |
| deploy-setup 模板识别验收 | 部署链路 | 确认收敛目标真正达成 |

**前瞻性**
- 将“部署模板能识别”纳入验收标准，是面向自动化交付的必要升级；未来类似项目也应优先以交付结构而非本地开发结构作为验收基线。

**结论**
- 推荐方案：**把首页可访问、API 可访问、WS 可访问、deploy-setup 可识别，作为同等级验收条件。**

---

### 2.4 推荐总体方案
本项目的新目标架构应定义为：

> 使用 FastAPI 作为唯一对外服务入口，继续承载 `/api` 与 `/ws`；保留 Vue 前端工程作为独立开发资产，但其 build 产物在交付时并入后端镜像，由同一容器统一提供静态页面与资源；整个系统对外只有一个服务、一个端口、一个 Dockerfile、一个 compose service、一个 deploy-setup APP_PORT。

## 三、实现路径（怎么办）

### 3.1 用户动线

#### 动线 1：最终用户访问系统
1. 用户打开域名；
2. 请求进入单一 FastAPI 服务；
3. FastAPI 返回 Vue 构建后的首页 `index.html`；
4. 浏览器加载 JS/CSS 静态资源；
5. Vue 页面初始化后请求 `/api/*` 获取基础状态；
6. Vue 页面建立 `/ws/events` 连接；
7. 页面开始实时展示监控状态、日志、连接状态。

#### 动线 2：用户执行监控操作
1. 用户在 Vue 页面中点击启动监控；
2. 前端调用 `/api/monitor/start`；
3. FastAPI 内部调用监控服务；
4. 监控服务产生运行状态、日志、行更新事件；
5. 事件通过 `/ws/events` 推送回浏览器；
6. 页面实时刷新表格、日志与状态面板。

#### 动线 3：用户刷新或直接访问前端子路由
1. 用户直接访问某个前端页面路径；
2. FastAPI 先判断是否命中 `/api` 或 `/ws`；
3. 若不是接口路径，则检查是否为真实静态文件；
4. 若不是静态文件，则回退返回 `index.html`；
5. Vue Router 在浏览器端恢复对应页面。

### 3.3 数据流

#### 页面访问流
```text
浏览器 GET /
  -> FastAPI 返回 index.html
  -> 浏览器加载 /assets/*
  -> Vue 应用启动
```

#### API 数据流
```text
Vue 页面动作
  -> HTTP /api/*
  -> FastAPI Router
  -> Monitor/Config Service
  -> JSON 响应返回页面
```

#### WebSocket 数据流
```text
Vue 建立 /ws/events
  -> FastAPI WebSocket endpoint
  -> ConnectionHub 注册连接
  -> MonitorService/Event Hub 广播事件
  -> 浏览器实时接收 row/log/connection/running
```

#### 前端路由回退流
```text
浏览器直接访问 /dashboard 或其他前端子路径
  -> FastAPI 发现非 /api、非 /ws
  -> 若静态文件不存在
  -> 返回 index.html
  -> Vue Router 接管页面
```

### 3.4 最小模块拆解对应的实施组织

#### 模块 1：静态资源挂载
- 目标：让 FastAPI 可直接提供 Vue build 产物；
- 输入：前端构建目录；
- 输出：浏览器可获取 JS/CSS/图片等静态资源。

#### 模块 2：前端路由回退到 `index.html`
- 目标：兼容 SPA history 路由；
- 输入：浏览器直接访问前端子路径；
- 输出：返回 `index.html` 而不是 404。

#### 模块 3：`/api` 与 `/ws` 保留
- 目标：保持现有后端协议边界不变；
- 输入：HTTP API 与 WS 请求；
- 输出：原有 API/WS 能力继续工作。

#### 模块 4：前端构建产物并入后端镜像
- 目标：形成单镜像交付物；
- 输入：Vue build 产物；
- 输出：镜像中包含页面静态资源与 Python 服务代码。

#### 模块 5：deploy-setup 兼容改造
- 目标：使自动部署工具只需要理解单服务结构；
- 输入：Dockerfile、端口、启动命令、compose service；
- 输出：deploy-setup 可识别、可生成、可安全部署的单服务产物。

## 四、验收标准

### 4.1 功能验收
- [ ] 浏览器打开域名后，直接看到 Vue 页面，而不是后端 JSON 或空白页。
- [ ] Vue 页面静态资源可正常加载，无需依赖第二个前端服务。
- [ ] `/api/health` 正常返回健康检查结果。
- [ ] `/ws/events` 可正常建立连接并收到基础连接/事件消息。

### 4.2 架构验收
- [ ] FastAPI 仍然保留 API 与 WebSocket，未因单服务收敛而被删除或替换。
- [ ] Vue 前端仍然保留为独立工程，只是交付形态改为 build 后并入后端镜像。
- [ ] 对外只有一个服务、一个端口、一个 Dockerfile、一个 compose service、一个 deploy-setup APP_PORT。
- [ ] `/api` 与 `/ws` 路由空间不会被前端静态页面错误覆盖。
- [ ] 前端 history 路由访问时，未命中静态资源的路径可正确回退到 `index.html`。

### 4.3 部署验收
- [ ] deploy-setup 可以识别当前项目为单服务结构，而非双服务结构。
- [ ] deploy-setup 可以安全生成单服务部署产物，不产生歧义端口或多 service 配置。
- [ ] 单服务镜像同时包含后端运行代码与前端构建产物。
- [ ] 自动部署链路仅需配置一个 APP_PORT 即可对外暴露完整系统。

### 4.4 边界验收
- [ ] 本次方案不是回退 Tkinter。
- [ ] 本次方案不是放弃 Vue。
- [ ] 本次方案仅改变部署与交付形态，不改变核心业务策略方向。

---

## 五、阶段结论

1. 旧的“前后端双服务”思路在技术上可行，但与 deploy-setup 当前单服务模板不兼容，因此不能作为本项目当前交付基线。
2. 最合理的收敛路径不是放弃 Vue，而是保留 Vue 工程，改为 build 后由 FastAPI 在同一容器中统一提供页面、API 与 WebSocket。
3. 本项目中“单服务”的精确定义必须同时满足：一个服务、一个端口、一个 Dockerfile、一个 compose service、一个 deploy-setup APP_PORT。
4. 技术边界必须保持清晰：`/api` 与 `/ws` 继续保留，前端仅改变交付形态；前端路由通过 `index.html` 回退实现浏览器直达访问。
5. 验收不能只看本地运行，还必须把 deploy-setup 的可识别性与安全产物生成纳入正式验收标准。
